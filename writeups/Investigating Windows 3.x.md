## Scenario

This room is the third in the Investigating Windows series and by far the deepest. Where 1.0 gave a brief look at a compromised endpoint and 2.0 pushed a little harder, 3.x drops you into a full forensic investigation: find the artifacts, decode the payloads, trace the attacker's movement and map everything back to MITRE ATT&CK.

Overall? thirty questions. Some chain together cleanly. Others had me second-guessing myself and going back to scratch. I'm keeping all of that in here; the wrong turns are part of the process,and honestly, they're where the real learning happens.

## Investigation

### Here I am looking for the registry key with the encoded payload.
First stop: Autoruns. There was a shortcut on the Desktop, launched it and started scanning the entries. One jumped out immediately: an entry named **Updater** with an Image Path that had no business being there.

Double-clicked Updater and Registry Editor opened automatically. Navigating into `CurrentVersion`, I found a key called `Debug` sitting there with a Data field packed with what was clearly encoded content. There was our payload.

https://github.com/user-attachments/assets/4be88e36-d0b8-421e-9a1a-92e8bd6927a8
https://github.com/user-attachments/assets/ede4b0d7-4c08-4c5e-b67a-ead29c35c111

----------

### Here I am looking for the rule name for a run key generated by Sysmon

Opened Event Viewer and expanded: `Application and Services Logs > Microsoft > Windows > Sysmon > Operational`

Right-clicked the first event, selected **Find**, typed `Updater` and clicked **Find Next**. The General tab of the matching event had the rule name mapped directly, `T1547.001`. Sysmon was already tagging it by MITRE technique. That's exactly why Sysmon config matters.

----------

### I am now looking into tactics that are classified with this MITRE ATT&CK ID?

I navigated to [attack.mitre.org](https://attack.mitre.org/) and looked up **T1547.001 - Boot or Logon Autostart Execution: Registry Run Keys / Startup Folder**.

Two tactics listed: **Persistence** and **Privilege Escalation**. The attacker planted this Run key to survive reboots. Depending on the execution context, it could also serve as a privilege escalation vector, which is why it maps to both.
https://github.com/user-attachments/assets/73bc793e-47df-44c5-b932-bd3a38f07cb6
https://github.com/user-attachments/assets/4609c0db-f6a0-405d-96e1-4a4bb4100a70

----------

### I am now looking for UTC time for the Sysmon event

Still in the same Sysmon event from 1.2. The timestamp caught my attention, just after 1 AM UTC. Classic off-hours attacker behaviour. Nobody's watching the SIEM at that time. Or so they hoped.

----------

### Now, I am digging System Monitor looking for the Sysmon Event ID and Event Type here

Same event. Event ID 13 in Sysmon specifically tracks **registry value modifications** which is precisely what you'd see when an attacker writes an encoded payload into a Run key. If you ever want to hunt this type of persistence, filtering Sysmon logs for Event ID 13 targeting `CurrentVersion\Run` paths is a solid starting point.


----------

### I had to decode the payload using CyberChef and what service will the payload attempt to start

Back to Registry Editor. Clicked on the `Debug` key, copied the full Data field content, and took it to CyberChef. Built this recipe:

`From Base64` > `Remove null bytes` > `Generic Code Beautify`

Hit **Bake!** and a full PowerShell script came out the other side. The very first line made the intent clear:

```powershell
sc.exe start Fax;

```

----------

### In this instance, the payload attempts to open a local port. What is the port number? It is my job to find out

Found in the same decoded output:

```powershell
$FTPPort = "9299";

```

The script sets up a TCP connection to localhost on port 9299, essentially staging a local listener that the Fax service will communicate through.

----------

### What process does the payload attempt to terminate? Let us explore

Further down in the decoded script:

```powershell
kill (Get-Process FXSSVC).Id -force;

```

FXSSVC is the Fax Service executable. The attacker starts the service, uses it, then kills it which forms a clean-up move to reduce traces. There was also a second encoded string buried inside the CyberChef output. Copied it, cleared the input panel, pasted the new string, ran the same recipe and baked again.

----------

### I had to find out what DLL file does the payload attempt to remove, follow along

Found in the second decoded output:

```powershell
Remove-Item -path 'C:\Windows\System32\ualapi.dll';

```

`ualapi.dll` is associated with the User Access Logging service. Removing it is a deliberate anti-forensics step, this meant that the attacker is cleaning up the crime scene.

----------

### Here I had to find the Windows Event ID associated with this service

Back to Event Viewer. This time I navigated to: `Application and Services Logs > Microsoft > Windows > PrintService > Admin`

Analysed the General tab of the relevant event - Event ID **823** showed up, associated with the Fax/Print Spooler service being manipulated. The hint pointed toward the DLL's link to Print Spooler or Fax. That guided me to the right log channel.

----------

### I had to go through the crime scene looking for what could be listed as the New Default Printer

Found in the same Event Viewer entry from 1.10. The event logged a printer being set as the new default and the name was **PrintDemon**. That name isn't random. PrintDemon is a known Windows Print Spooler privilege escalation vulnerability. The attacker weaponised it here.

----------

### I had to dig for the process is associated with this event   (spoolsv.exe)

Switched to Procmon. Hit `Ctrl+F`, typed `PrintDemon`, clicked **Find Next**. The associated process came back as `spoolsv.exe` which is the Windows Print Spooler service process. This is the target of the PrintDemon exploit.

----------

### What is the parent PID for the above process? Let's look this up

Right-clicked the `spoolsv.exe` event in Procmon and selected **Properties**. Navigated to the **Process** tab — parent PID was `620`.

----------

### Here I am looking for the PID of the process running the encoded payload

This one required a Procmon filter. Went to **Filter > Filter**, disabled existing conditions, then set:

`Detail` > `contains` > `-enc`

Clicked **Add** > **Apply**. The `-enc` flag is PowerShell's encoding switch. A classic sign of an obfuscated payload being executed. The filtered results pointed to PID **3088**.

----------

### I have to do more decoding, in this instance? decoding the payload and its visible partial path

Went back to the decoded output from 1.8, there was another encoded string I hadn't touched yet. Ran it through CyberChef with the same recipe and found the partial path:

```
/admin/get.php

```

I copied the full decoded output and saved it, I had a feeling it would keep giving answers.

----------

### What attack framework was used? What is the name of the variable?

Opened Explorer and navigated to `This PC > Documents > 20210121`. Three files in the folder, the third one (ending in `7954`) contained a reference to `Invoke-PSInject`.

My first instinct was to just Google the function name. The results confirmed it's a module from **PowerShell Empire** which is a post-exploitation framework used for lateral movement and persistence.

Then I searched specifically for "Customizing PowerShell Empire" and found documentation showing that the default communication profile uses a set of default file path names, which matched the `/admin/get.php` path from the previous task exactly. The variable name in the source: **DefaultProfile**.

----------

###  What other file paths am I likely to find in the logs?

This one tripped me up. I found Empire documentation listing default paths as `/news.asp` and `/login/process.jsp`. I submitted that, wrong. Went back to the decoded output from 1.6 looking for the files directly. Couldn't find them.

Eventually I tried swapping the extensions to `.php` and it worked. Something I grasped from my mentor that you brute force and think outside the box often times. I can't fully explain why the room expected `.php` extensions specifically rather than the defaults listed in the docs, but the lesson here is: when documentation and evidence diverge, test the evidence first.

----------

###  What is the MITRE ATT&CK URI for the attack framework I wonder! Let's look

Searched MITRE ATT&CK for "Empire" and found the software entry. The S-number designates it as a Software entry in the ATT&CK catalogue. Empire is well documented there with all the techniques it's been observed using.

----------

###  What was the FQDN of the attacker machine?

There was still one more encoded string in the CyberChef output I hadn't decoded. Pasted it into the Input panel with a simpler recipe this time:

`From Base64` > `Remove null bytes`

The output gave an IP: `http://34.245.128.161:9001`

Then in Procmon, I enabled **Show Network Activity**, found the first event matching that IP, right-clicked > **Properties**, and copied the full path, which surprisingly resolved to the Amazon EC2 FQDN above. The attacker was running their C2 on AWS infrastructure.

----------

### Let us look into other process the attacker might have connected to their machine

Back in Procmon. Hit `Ctrl+F`, pasted the FQDN from 1.19, clicked **Find Next**.

I ran PowerShell from the filter (`Process Name is powershell.exe → Exclude`), applied it and hit the search. The next result came back as **Explorer.exe**. The attacker had injected into the Explorer process, which lines up with what's coming in second last task. How I knew what to run? I can't tell but it worked. Maybe it was a hunch!

----------

### I am now looking into the PID for this process

Know what I did? I went two tasks back and there it was in the same Procmon event.

----------

### What was the path for the first image loaded for the process? Let us look


Set two filters in Procmon:

-   `Operation is Load Image → Include`
-   `PID is 2684 → Include`

Applied both and looked at the first result - `mscoree.dll`, the Microsoft .NET Runtime Execution Engine. Its presence here is a signal that .NET-based injection code was being loaded into the Explorer process.

----------

### Now, it is time to see what Sysmon event was generated between these 2 processes as well as the Event ID

I already knew from the Sysmon documentation that Event ID 8 tracks `CreateRemoteThread`. This is an event that fires when one process creates a thread inside another. That's textbook process injection. Learnt that as I was working my way around this here.

Went to Event Viewer: `Application and Services Logs > Microsoft > Windows > Sysmon > Operational`

Filtered for Event ID 8, sorted by Date and Time ascending. Found the answer in the third event which is a thread being created from `powershell.exe` into `Explorer.exe`.

----------

### OK, time to look for the UTC time for the first event between these 2 processes

Found in the same Sysmon event from a task I did earlier. Note that this timestamp is _"before"_ the Run key was written `01:08:13`..... the injection happened first, then persistence was established. The attacker's sequence is becoming clear.

----------

### I looked for the first operation listed by the 2nd process starting at this time

Found in Procmon. The first operation logged for the Explorer.exe process at that timestamp was **Create Thread** 
Procmon caught the moment and went on to inject the thread that spun up inside Explorer.

----------

### Time to look for the full registry path queried by the attacker to get information about the victim now and we found it, this is how

Followed the hint and filtered Procmon for events starting at `1/21/2021 5:07`. Spotted an event that flagged as a buffer overflow, right-clicked and opened **Properties**. The path being queried was the Windows release ID registry key. The attacker was doing system reconnaissance, checking the OS version of the machine they'd just landed on.

----------

### What is the name of the last module in the stack with a successful result I wonder, let us look into it rightaway

From the same event, clicked the **Stack** tab. The last module listed with a successful result was labelled `<unknown>`, meaning the code running there had no identifiable module backing it. 

----------

### I found myself asking the question "what module within the attack framework was most likely used between the 2 processes?"

Already found this during a task I cleared earleir when I was analysing the files in the Documents folder. `Invoke-PSInject` is a PowerShell Empire module that injects a PowerShell runspace into another process, which is exactly what we observed between `powershell.exe` and `Explorer.exe`.

----------

### What is the MITRE ID for this technique you wonder? Let's pay MITRE a visit and Google

Searched MITRE ATT&CK for `Invoke-PSInject` and it maps to **T1055 - Process Injection**. The attacker used Empire's injection module to move from a PowerShell process into Explorer, gaining a foothold inside a trusted system process to evade detection.

----------

## Attack Chain Summary

Piecing it all together, here's what happened on this endpoint:

```
PowerShell (PID 3088) executed encoded payload
        ↓
Invoke-PSInject injected into Explorer.exe (PID 2684)
        ↓
CreateRemoteThread event fired — 2021-01-22 01:07:06 UTC
        ↓
Fax service started → TCP listener on port 9299
        ↓
C2 beacon to ec2-34-245-128-161.eu-west-1.compute.amazonaws.com:9001
        ↓
PrintDemon exploit → spoolsv.exe abused as New Default Printer
        ↓
Registry Run key written for persistence (T1547.001) — 01:08:13 UTC
        ↓
FXSSVC killed + ualapi.dll removed to cover tracks

```

## Key Takeaways

This room reinforced something that doesn't get said enough: **the tools only work if you know where to look first.** Autoruns flagged the persistence. Sysmon already had the MITRE tag. Procmon showed the injection happening in real time. CyberChef peeled the layers back. None of that is useful unless you know how the pieces connect.

Understanding that sequence changes how you think about incident response: you're not just looking for what changed, you're reconstructing _when_ and _in what order_.

Empire is a real framework used in real engagements. Recognising its default communication profile (`/admin/get.php`, `news.php`, `/login/process.php`) is a practical detection skill, not just a CTF answer.
