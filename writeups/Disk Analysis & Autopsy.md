#  TryHackMe - Disk Analysis & Autopsy

**Difficulty:** Medium **Category:** Digital Forensics (DFIR) **Tool:** Autopsy

----------

##  Overview

This investigation involves analszing a forensic disk image using Autopsy to uncover malicious activity, identify user behaviour, extract system artifacts and piece together what happened on a compromised machine.

The objective was to:

-   Identify system and user information
-   Uncover installed malicious tools
-   Trace user activity through artifacts
-   Locate and analyze evidence of exploitation attempts

----------

##  What Is Autopsy?

Autopsy is an open-source digital forensics platform used by investigators to analyse disk images and file systems.

It allows analysts to examine:

-   File system contents (including deleted files)
-   Registry hives
-   Browser history and bookmarks
-   User account information
-   Installed applications
-   PowerShell and command history
-   Email and communication artifacts
-   Keyword search across the entire image

**Why disk forensics matters in SOC/IR:** When a system is compromised, the disk contains a permanent record of activity - files created, programs executed, registry changes made and accounts accessed. Even deleted files often remain recoverable. Disk forensics allows investigators to reconstruct the full timeline of an attack long after it has occurred.

##  Key Forensic Concepts Covered

----------

###  Forensic Image Hashing

Before analysing any disk image, the first step is verifying its **integrity using a hash**.

A hash is a fixed-length fingerprint of a file or dataset generated by a mathematical algorithm (MD5, SHA1, SHA256).

**Why hashing matters in forensics:**

-   Confirms the image has not been altered since acquisition
-   Provides legal defensibility - you can prove the evidence was not tampered with
-   Allows comparison against known-good or known-bad databases

###  Operating System Artifacts

Autopsy automatically parses key Windows artifacts, including:

**Operating System Information:** Reveals the computer name, OS version, and install date. Establishing this baseline is critical before investigating user activity.

**User Accounts:** Windows stores local user account information in the SAM registry hive. Autopsy surfaces this as readable data, showing:

-   All account names
-   Last login timestamps
-   Account types

**From this investigation:** Eight user accounts were identified. Sorting by last login timestamp revealed which account was most recently active — a key detail for focusing the investigation.

----------

###  Network Configuration Artifacts

Identifying a machine's network identity is essential for correlating it with network logs, firewall data, and SIEM events.

**Standard approach — Windows Registry:** Network adapter configuration is stored at:

```
HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces
```

However, in this case the IP address was listed as `0.0.0.0`, meaning it was not available through this path.

**Alternative approach — Application Artifacts:** An unusual application, **Look@LAN**, was found installed on the machine. Look@LAN is a network monitoring tool that stores configuration in an `.ini` file within its program directory.

**What is an .ini file?** An initialization file stores configuration settings in plain text. Applications read these on startup to load their settings. Forensically, they can contain network configuration, user preferences, and connection history.

The `.ini` file contained both the IP address and MAC address of the machine.

**Network card identification:** The full name of the network interface card (NIC) was found in the registry at:

```
SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkCards
```

**Key lesson:** When standard artifact locations fail, pivot to application-generated artifacts. Installed tools often store system information in unexpected places.

----------

###  Web Artifacts — Bookmarks and Browser History

Web artifacts are among the most valuable sources of investigative intelligence.

**What browser artifacts reveal:**

-   Sites visited (timeline of activity)
-   Files downloaded
-   Searches performed
-   Bookmarked locations or resources
-   Evidence of attacker research (looking up tools, exploits)

**From this investigation:** A Google Maps bookmark was found containing GPS coordinates. This type of artifact can be significant in investigations involving physical locations, meeting points, or target reconnaissance.

Browser history was also used later to help identify malicious tools — checking what users had searched for and downloaded.

----------

###  User Profile Artifacts — NTUSER.dat and Wallpaper

Every Windows user has an `NTUSER.dat` file stored in their home directory. This file is the user's personal registry hive and contains settings, preferences, and history specific to that account.

**What NTUSER.dat contains:**

-   Recently opened files and folders
-   Desktop wallpaper path
-   Recently run programs
-   Typed paths in Explorer
-   Software settings specific to the user

**From this investigation:** NTUSER.dat was used to identify the wallpaper image path for each user. By examining the wallpaper images, one was found to contain the user's full name — a personal customization that became a forensic artifact.

----------

###  PowerShell History

PowerShell is a powerful scripting environment built into Windows. It is heavily used by both administrators and attackers.

**Where PowerShell history is stored:**

```
C:\Users\[username]\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

This file stores every command a user typed in PowerShell — unless the history has been deliberately cleared.

**Why this is critical in investigations:**

-   Reveals attacker commands run on the system
-   Shows file manipulations, downloads, and privilege escalation attempts
-   Can expose flags, credentials, or evidence of lateral movement

**From this investigation:** A user's PowerShell history revealed that a file on their desktop had been modified. The original content was recoverable through the history, revealing the original flag before it was changed.

A PowerShell script named `exploit.ps1` was also found on the same user's desktop — a privilege escalation exploit containing a message intended for the device owner.

----------

###  Credential Dumping Tools

Two password-related hack tools were found on the system:

----------

**Mimikatz:** Mimikatz is one of the most well-known post-exploitation tools in cybersecurity. It was developed by Benjamin Delpy and is capable of:

-   Extracting plaintext passwords from Windows memory (LSASS)
-   Dumping password hashes
-   Performing Pass-the-Hash and Pass-the-Ticket attacks
-   Extracting Kerberos tickets

It is heavily used by threat actors after gaining initial access to escalate privileges or move laterally across a network.

**Detection in this investigation:** A zip file (`mimikatz_trunk.zip`) was found in a user's Downloads folder. Additionally, a YARA rule file on the system was authored by **Benjamin DELPY (gentilkiwi)** — the creator of Mimikatz — further confirming its presence.

----------

**LaZagne:** LaZagne is an open-source credential recovery tool that extracts saved passwords from:

-   Browsers
-   Email clients
-   Databases
-   System applications
-   Wi-Fi configurations

Unlike Mimikatz which targets memory, LaZagne targets application-stored credentials.

**Detection in this investigation:** LaZagne was not found through standard artifact locations (browser history, downloads, installed programs). It was discovered through **Windows Defender scan history logs**.

**Where Defender stores its logs:**

```
C:\ProgramData\Microsoft\Windows Defender\Scans\History
```

The Defender logs showed alerts for both Mimikatz and LaZagne, providing the evidence needed to confirm both tools were present on the system.

**Key lesson:** When artifacts are not found through obvious channels, check security tool logs. Windows Defender, event logs, and antivirus history are underutilized but highly valuable forensic sources.

----------

###  YARA Rules

YARA is a tool used by malware researchers and SOC analysts to create rules that identify and classify malware based on patterns.

A YARA rule looks like this:

```
rule ExampleRule {
    meta:
        author = "Analyst Name"
        description = "Detects suspicious string"
    strings:
        $s1 = "malicious_string"
    condition:
        $s1
}
```

**From this investigation:** A `.yar` file was found on the system. The author field inside the file identified it as belonging to **Benjamin DELPY (gentilkiwi)** — the creator of Mimikatz — confirming the connection between the YARA file and the credential dumping tools found elsewhere on the system.

----------

###  Zerologon (MS-NRPC Exploit)

One of the users had downloaded an exploit targeting a **domain controller** using a vulnerability known as **Zerologon**.

**What is Zerologon (CVE-2020-1472)?** Zerologon is a critical vulnerability in Microsoft's Netlogon Remote Protocol (MS-NRPC). It allows an unauthenticated attacker on the network to:

-   Impersonate any computer on the domain
-   Gain domain administrator privileges
-   Fully compromise a Windows domain controller

It was rated **CVSS 10.0** — the maximum severity score — when disclosed in 2020.

**Why this matters:** The presence of a Zerologon exploit on a machine indicates the user was either planning to attack the domain controller or had already done so. Combined with the credential dumping tools found elsewhere, this paints a picture of a user actively attempting to escalate privileges and move laterally toward the domain's core infrastructure.

**Detection in this investigation:** A keyword search for "zerologon" returned a hit for a zip file in a user's downloads folder. Although the file appeared to have been deleted, Autopsy recovered the reference, confirming it had been present on the system.

##  Key Lessons Learned

**1. Always verify image integrity first.** Hash verification is non-negotiable in forensics. It establishes evidence integrity before anything else.

**2. Standard artifact locations are not always sufficient.** Network configuration was not available in the registry. The answer came from an application artifact — a configuration file from a third-party tool. Analysts must know when to pivot.

**3. Windows Defender logs are an underutilized forensic source.** Both malicious tools in this investigation could have gone undetected without checking Defender's scan history. Security tool logs should always be part of the investigation checklist.

**4. PowerShell history is a goldmine.** Attackers frequently use PowerShell. Unless history is deliberately cleared, every command is logged and recoverable.

**5. Deleted files leave traces.** The Zerologon exploit archive had been deleted, but Autopsy still surfaced evidence of its presence. Deletion is not the same as erasure.

**6. Everything is connected.** The YARA file, Mimikatz zip, Defender alerts, and author attribution all pointed to the same person and the same toolkit. Building a complete picture requires correlating all available evidence.

----------

##  Conclusion

This investigation reinforced how disk forensics can reconstruct a detailed picture of attacker activity — even across multiple users, tools, and techniques. It demonstrated that evidence is rarely found in just one place, and that the most valuable artifacts are often the ones you have to actively look for.

From a SOC perspective, this type of investigation represents what comes after the alert — the deeper dive that turns an indicator into a full incident narrative.
